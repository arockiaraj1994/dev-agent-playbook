MCP tools at a glance

The server exposes a small, AI-agent-friendly tool surface. Tell your editor about start_task first — it returns guardrails + the matched workflow + the next calls to make, all in one round-trip.

ToolWhen the agent should call it
start_task(project, task)First call for any task. Free-form sentence in; bootstrap bundle out.
list_projects()Discovery when project name unknown.
get_index(project)Browse the trigger map without committing to a task.
get_guardrails(project)Re-read always-on rules mid-task.
get_agents_md(project)Identity / behavior doc.
get_architecture(project, name?)Overview, or a specific ADR.
get_language_rules(project, language, doc?)Per-language standards / testing / anti-patterns.
get_pattern(project, pattern)Implementing a known pattern.
get_skill(project, skill)Executing a verb-noun playbook.
get_workflow(project, name)Working a task-driven flow (new-feature, bug-fix, security-fix, refactor).
get_gate(project, name?)Verifying done-ness; lists or describes verify-*.sh scripts.
list_rule_docs(project, doc_type?)Discovery within a project.
search_rules(query, …)BM25 fallback when nothing else matches.

Project layout

Each project lives next to mcp/ in its own directory with a fixed layout. The validator enforces it; the loader indexes by path; INDEX.md is auto-generated.

Before writing a new doc, check what's already there: Browse existing rules → — and copy from a live project (apache-camel/, baton-sso-config/, integration-manager/) for a known-good baseline.

Every project directory follows this shape. Files marked required are checked by scripts/validate-rules.py; the validator fails CI if any are missing.

<your-project>/
├── README.md                          ← humans
├── AGENTS.md                          ← required — AI entry point
├── INDEX.md                           ← auto-generated trigger map (do not hand-edit)
├── core/
│   ├── guardrails.md                  ← required — always-on MUST / MUST NOT
│   ├── definition-of-done.md          ← required — tests + lint + security gates
│   └── glossary.md                    ← required — domain terms
├── architecture/
│   ├── overview.md                    ← required — system overview
│   ├── decisions/                     ← ADRs (one .md per decision)
│   └── diagrams/                      ← optional images
├── languages/
│   └── <lang>/                        ← java | typescript | python | kotlin | sql | shell-yaml
│       ├── standards.md               ← at least one languages/<lang>/standards.md required
│       ├── testing.md
│       └── anti-patterns.md
├── patterns/
│   └── <name>.md                      ← canonical noun-named patterns
├── skills/
│   └── <action>.md                    ← verb-noun playbooks
├── workflows/                         ← all four required
│   ├── new-feature.md
│   ├── bug-fix.md
│   ├── security-fix.md
│   └── refactor.md
└── gates/
    ├── README.md                      ← required — what each gate enforces
    └── scripts/
        └── verify-<lang>.sh           ← executable (chmod +x)

The loader infers doc_type from path. Anything outside the recognized folders lands in other and the validator fails.

Frontmatter that wires the agent

The loader reads optional YAML frontmatter. Two fields are load-bearing for AI workflows: triggers (used by start_task matching and the INDEX.md generator) and see_also (rendered as a ## Next Calls section in tool responses).

---
title: Workflow — Bug fix
description: Reproducing, isolating, and fixing a defect in an existing route.
triggers: [bug, fix a bug, debug a route, route is failing]
gates: [verify-java]
see_also: [skill:debug-route, pattern:error-handling, language:java/testing]
---

# Workflow — Bug fix

## Steps
1. Reproduce first.
2. ...
  • triggers: — natural-language phrases the agent will match against the user's task. Workflows and skills should set this.
  • see_also: — entries of the form <kind>:<name> where kind ∈ pattern, skill, workflow, gate, language, architecture. Each entry becomes a clickable next call in the tool response.
  • language: — set automatically for languages/<lang>/*.md; can be set explicitly elsewhere.
  • gates: — on workflows, names the verify-*.sh script(s) that close out the task.

Validate & reload

After adding or editing docs, regenerate INDEX.md and run the validator. CI runs --check and fails the build if anything is missing or stale.

python scripts/validate-rules.py --regen-index   # rewrite INDEX.md
python scripts/validate-rules.py --check         # full validation + INDEX freshness
chmod +x <project>/gates/scripts/verify-*.sh     # if you added a new gate

The server reads files once at startup. After any structural change, restart so the new docs (and any new MCP tool wiring) take effect.

Watch the Searches → Zero-result tab — every entry there is a doc your team is asking for. Pick one, write it as a workflow, skill, or pattern, and you'll see the badge clear once the agent finds it.
