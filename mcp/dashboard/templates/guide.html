MCP tools at a glance

Five read-only tools, all playbook_-namespaced. playbook_start_task is the only coding entry point - it returns identity + guardrails + optional requirement + matched workflow + a single ## Next Calls block of ready-to-paste playbook_get_doc(…) calls.

ToolWhen the agent should call it
playbook_start_task(project, task, requirement?)First call for any coding task. Free-form sentence in; identity + guardrails + matched workflow out. Pass requirement=ST-101 (or a PRD id) for a server-side tree walk.
playbook_get_doc(kind, project, name?, section?, depth?)Fetch any standards or requirement doc. kind: agents, guardrails, architecture, language, pattern, skill, workflow, gate, requirement. Corpus is implied by kind.
playbook_search_docs(project, query?, doc_type?, corpus?)Discovery. Omit query to list docs + triggers; pass query to search. corpus: standards (default), requirements, or all.
playbook_list_requirements(project, …)Catalogue PRDs/stories (ids + summaries, never bodies).
playbook_start_requirement(project, intent, …)PM authoring bootstrap - template + glossary + guardrails + next id + suggested targets:. Not for coding (use playbook_start_task).

Every doc response that declares see_also: or targets: ends with ## Next Calls rendered as playbook_get_doc(kind=…) - that is how agents chain without guessing paths. project is always required: the basename of the user's workspace directory.

Corpus layout

Docs live under two roots next to mcp/: standards/ (HOW) and requirements/ (WHAT). The validator enforces the shapes; the loader indexes by path; INDEX.md is auto-generated in each project.

Browse health on the dashboard: Standards → and Requirements →. Copy from a live project (standards/nexre/, standards/apache-camel/, requirements/nexre/).

Standards - standards/<project>/

Files marked required are checked by scripts/validate-rules.py --corpus standards.

├── architecture/
│   ├── overview.md                    ← required - system overview
│   ├── decisions/                     ← ADRs (one .md per decision)
│   └── diagrams/                      ← optional images
├── languages/
│   └── <lang>/                        ← java | typescript | python | kotlin | …
│       ├── standards.md               ← at least one languages/<lang>/standards.md
│       ├── testing.md
│       └── anti-patterns.md

│   └── <name>.md                      ← canonical noun-named patterns

Requirements - requirements/<project>/

Same project slug as standards (e.g. nexre). Stories live under their PRD folder - path is the parent link, no parent: field needed. Author via playbook_start_requirement or copy mcp/templates/PRD.md / STORY.md.

requirements/<your-project>/
├── AGENTS.md                          ← short PM-facing identity (optional but useful)
├── INDEX.md                           ← auto-generated
├── workflows/
│   ├── write-prd.md                   ← matched by playbook_start_requirement(type=prd)
│   └── write-story.md                 ← matched by playbook_start_requirement(type=story)
└── PRD-001-<slug>/
    ├── prd.md                         ← id: PRD-001 in frontmatter
    └── stories/
        ├── ST-101-<slug>.md           ← targets: [pattern:…, skill:…] → Next Calls
        └── ST-102-<slug>.md

The loader infers doc_type from path. Anything outside the recognized folders lands in other and the validator fails. Standards are boot-cached; requirements use a TTL cache (reload from the Requirements page or restart).

Frontmatter that wires the agent

The loader reads optional YAML frontmatter. Load-bearing fields:

  • triggers: - phrases playbook_start_task matches against the user's task (workflows / skills).
  • see_also: - standards chain. Entries <kind>:<name> become playbook_get_doc(…) in ## Next Calls.
  • targets: - stories only. Same vocabulary as see_also; the WHAT→HOW bridge into standards (merged into playbook_start_task's Next Calls when requirement= is set).
  • id: / status: / priority: - requirements. Status is draft | approved | shipped (drafts soft-fail validation; approved/shipped hard-fail).
  • gates: - on workflows, names the verify-*.sh script(s) that close the task.
---
title: Workflow - Bug fix
description: Reproducing, isolating, and fixing a defect.
triggers: [bug, fix a bug, debug a route]
gates: [verify-java]
see_also: [skill:debug-route, pattern:error-handling, language:java/testing, core:guardrails]
---

# Workflow - Bug fix

## Steps
1. Reproduce first.
2. ...

Valid kinds: tool, pattern, skill, workflow, gate, language, architecture, core, agents, requirement. Valid tool: names: playbook_start_task, playbook_get_doc, playbook_search_docs, playbook_list_requirements, playbook_start_requirement (pre-0.7.0 names like start_task still render). Unrecognized entries are a validation error - they would otherwise render as nothing and silently break the chain.

Validate & reload

After adding or editing docs, regenerate INDEX.md and run the validator. CI splits standards (blocking) from requirements (continue-on-error for drafts).

# From repo root (or via: cd mcp && uv run dev.py - regenerates + validates before start)
python scripts/validate-rules.py --corpus all --regen-index
python scripts/validate-rules.py --corpus standards --check
python scripts/validate-rules.py --corpus requirements --check
chmod +x standards/<project>/gates/scripts/verify-*.sh   # if you added a gate

Standards are loaded once at boot - restart the server after structural changes. Requirements use a TTL cache; admins can force a reload from the Requirements page without a full restart.

Watch the Searches → Zero-result tab - every entry is a doc your team is asking for. Write it as a workflow, skill, pattern, or story, and the badge clears once the agent finds it.
